# R3: 업계/직업 구조 -- 한국 IT 업계 + 헌터물 조직 체계

**신뢰도**: MEDIUM
**근거**: 웹 검색 불가로 학습 데이터(~2025.05) 기반. 한국 IT 업계 구조는 공개 보도/보고서 기반 사실, 헌터물 조직 구조는 주요 작품 다수에서 교차 확인. 미확인 항목 약 15%.

## 1. 한국 SI/SM 업계 계층 구조

### 1-1. 다단계 하도급 피라미드

| 계층 | 대표 기업 유형 | 역할 | 인력 규모 |
|------|-------------|------|----------|
| **원청 (Tier 0)** | 삼성SDS, LG CNS, SK C&C, 현대오토에버 | 대기업 계열 SI사. 수주/기획/PM | 수천~만 명 |
| **1차 협력사 (Tier 1)** | 중견 SI: 한화시스템, 롯데정보통신 등 | 설계/핵심 개발 | 수백~천 명 |
| **2차 협력사 (Tier 2)** | 중소 SI/SM 업체 | 실제 코딩/유지보수 | 수십~백 명 |
| **3차 협력사 / 프리랜서** | 영세 업체, 1인 개발자 | 단순 코딩, 테스트, 운영 | 1~10명 |

**핵심 갈등 구조**:
- 원청이 수주한 프로젝트가 2~3차까지 재하도급되며 단가가 30~50% 깎임
- 실제 개발은 2~3차에서 하지만, 산출물 저작권과 성과는 원청에 귀속
- "몸값"(man/month) 기반 계약: 개발자 1인의 월 단가로 거래, 역량보다 머릿수가 중시됨
- 프리랜서 단가: 주니어 300~400만 원/월, 시니어 500~700만 원/월 (2024 기준 시장가)

### 1-2. 주인공 안강해의 위치: 중소 SI 4년차 백엔드 개발자

**2차 협력사 소속의 현실**:
- 원청 사무실에 상주 파견 (자사 사무실에 거의 안 감)
- 원청 직원과 같은 공간에서 일하지만, 식당/복지 시설 이용 제한 (일부 회사)
- 프로젝트 완료 시 다른 현장으로 이동 ("파견 로테이션")
- 야근/주말 근무가 잦지만, 원청 직원보다 낮은 대우
- 자사 소속감 희박: "나는 어디 소속인가" 정체성 혼란

**감정적 소재**:
- 원청 사원이 "협력사분"이라 부를 때의 보이지 않는 벽
- 같은 일을 하되 연봉/복지/성과 인정의 격차
- 기술 탈취: 개발한 솔루션이 원청 명의로 등록됨
- 프로젝트 종료 후 "칼 퇴근" 통보 (계약 미연장)

### 1-3. IT 업계 직급/권력 구조

| 직군 | 계층 | 권한 | 갈등 포인트 |
|------|------|------|-----------|
| **PM/PL** | 프로젝트 관리 | 일정/인력 배분, 요구사항 정의 | 비현실적 일정 지시, 기술 이해 부족 |
| **개발자** | 실행 | 설계/구현, 기술 결정 | 일정 압박 vs 품질, 기술 부채 |
| **QA** | 검증 | 테스트/결함 관리 | 개발팀과의 긴장 관계 |
| **영업/기획** | 수주 | 고객 접점, 계약 | 기술적으로 불가능한 약속 |
| **CTO/아키텍트** | 기술 리더십 | 기술 스택 결정, 표준 수립 | 실무와 괴리, 레거시 고집 |

**갑을 관계 체인**: 고객사 → 원청 영업 → 원청 PM → 협력사 PM → 개발자. 위에서 내려온 압박이 하단으로 집중됨.

## 2. 백엔드 개발자 커리어 패스

### 2-1. 경력 단계

| 연차 | 직급/호칭 | 역할 | 연봉 범위 (2024 기준) |
|------|----------|------|---------------------|
| 0~2년 | 주니어 | 기능 구현, 버그 수정, 코드 리뷰 수동적 참여 | 3,000~4,500만 원 |
| 3~5년 | 미드레벨 | 모듈 설계, API 설계, 성능 최적화, 후배 지도 | 4,500~6,500만 원 |
| 5~8년 | 시니어 | 아키텍처 설계, 기술 의사결정, 장애 대응 리드 | 6,500~9,000만 원 |
| 8~12년 | 리드/팀장 | 팀 관리, 기술 로드맵, 채용 면접 | 8,000~1.2억 원 |
| 12년+ | CTO/VP of Eng | 기술 전략, 조직 구조, 경영 참여 | 1억~2억+ |

**안강해(4년차)의 위치**: 미드레벨 진입기. 기본기는 갖추었으나 아키텍처 결정권은 없음. SI 4년차는 스타트업/대기업 이직 결심의 분기점.

### 2-2. 기술 스택과 서사 소재

| 기술 영역 | 구체 기술 | 서사적 활용 |
|----------|----------|-----------|
| **언어** | Java/Spring, Python, Go, Node.js | 스킬 코드의 "언어"로 치환 |
| **DB** | MySQL, PostgreSQL, Redis, MongoDB | 스킬 데이터 구조, 인덱싱 = 스킬 검색 |
| **인프라** | AWS, Docker, Kubernetes, CI/CD | 던전 = 서버, 레이드 = 배포 |
| **디버깅** | 로그 분석, 프로파일링, APM 도구 | 스킬 코드 디버깅의 핵심 |
| **API** | REST, GraphQL, gRPC | 스킬 간 연동, 파티 시너지 |
| **보안** | SQL Injection, XSS, 인증/인가 | 시스템 취약점 공격/방어 |

### 2-3. 개발자 전문 용어 (서사 활용 가능)

| 용어 | 실제 의미 | 소설 내 치환 |
|------|----------|-------------|
| **디버깅** | 코드 오류 찾아 수정 | 스킬 코드의 버그를 찾아 강화 |
| **리팩토링** | 기능은 유지, 구조 개선 | 스킬 재구성 (같은 효과, 더 효율적) |
| **컴파일** | 소스코드 → 실행 파일 변환 | 스킬 코드 편집 → 발동 가능 상태로 변환 |
| **런타임 에러** | 실행 중 발생하는 오류 | 스킬 발동 중 폭주/역효과 |
| **메모리 누수** | 사용 후 해제 안 된 메모리 | 마나/기 누수, 지속 소모 |
| **스택 오버플로** | 재귀 호출 과다 | 스킬 중첩 발동 시 과부하 |
| **패치** | 결함 수정 업데이트 | 스킬 코드 일부 수정 |
| **API 콜** | 외부 서비스 호출 | 다른 각성자 스킬 연동 |
| **레거시 코드** | 오래된 유지보수 어려운 코드 | 고대 스킬, 해독 불가한 코드 |
| **깃(Git) 커밋** | 변경사항 저장 | 스킬 수정 확정 (롤백 가능) |
| **롤백** | 이전 버전으로 되돌림 | 스킬 수정 실패 시 원복 |
| **샌드박스** | 격리된 테스트 환경 | 스킬 테스트 공간 (실전 영향 없음) |

## 3. 헌터물 장르 조직 구조

### 3-1. 표준 헌터 등급 체계

| 등급 | 비율 (전체 각성자 중) | 전투력 | 사회적 위상 |
|------|---------------------|--------|-----------|
| **S급** | 0.01% 미만 | 국가 전략 자산 | 재벌급 영향력, 정치적 발언권 |
| **A급** | 0.1~0.5% | 고난도 레이드 핵심 | 상류층, 대형 길드 간부 |
| **B급** | 1~3% | 중급 던전 공략 주력 | 중상류, 안정적 고소득 |
| **C급** | 5~10% | 일반 던전 운용 | 중산층, 직장인 수준 |
| **D급** | 15~20% | 저급 던전, 보조 역할 | 하위, 위험 대비 저수익 |
| **E급** | 30~40% | 전투 부적합, 비전투 계열 | 각성자이나 사실상 일반인 대우 |
| **F급** | 나머지 | 측정 불가 / 비각성 수준 | 각성자 취급 못 받음 |

### 3-2. 헌터 협회 구조

| 조직 | 역할 | 권한 |
|------|------|------|
| **헌터 협회 본부** | 등급 측정, 라이선스 발급, 분쟁 조정 | 헌터 활동 전반 규제 |
| **등급 심사위원회** | 각성자 등급 판정, 재심사 | 등급 상향/하향 결정 |
| **던전 관리국** | 던전 발생 모니터링, 접근 통제 | 던전 공략 허가/배분 |
| **분쟁 조정위** | 길드 간 분쟁, 사망사고 조사 | 제재/자격 정지 |
| **국제 협력부** | 타국 협회와 공조 | 국제 레이드, 외국 헌터 관리 |

**갈등 소재**: 협회는 표면상 중립이나, 대형 길드(재벌 계열)의 로비에 좌우됨. 등급 판정 기준의 불투명성. F급은 시스템적 사각지대.

### 3-3. 길드 구조

| 계층 | 역할 | 구성원 |
|------|------|--------|
| **길드 마스터** | 최종 결정권, 대외 대표 | 1인 (보통 최고 등급자) |
| **부마스터** | 실무 총괄, 마스터 부재 시 대행 | 1~2인 |
| **레이드 지휘관** | 던전 공략 지휘, 전술 결정 | A~B급 고참 |
| **핵심 전투원** | 레이드 주력 딜러/탱커/힐러 | A~B급 |
| **일반 대원** | 던전 공략 참여, 소재 수집 | B~C급 |
| **보조 인력** | 물류, 정보, 행정, 비전투 지원 | C~E급 |
| **수습/견습** | 신입, 훈련 중 | D~E급 |

**5대 길드 (작품 내 설정 가능 구조)**:

| 길드명 (가칭) | 배경 | 특성 |
|-------------|------|------|
| 재벌 계열 길드 | 대기업 자본 | 자원 풍부, 관료적, 정치적 |
| 실력주의 길드 | S급 독립 창설 | 전투력 위주, 소수정예 |
| 정부 직속 길드 | 국가 운영 | 공무원 체계, 의무 임무 |
| 외국 자본 길드 | 다국적 | 한국 자원 착취 논란 |
| 신생 벤처 길드 | 최근 설립 | 혁신적이나 불안정 |

### 3-4. 무능력자 부서 / F급 관리 조직

헌터물에서 F급 주인공이 소속되는 조직의 전형:
- **협회 부설 "지원과"**: 각성자이나 전투 불가한 인원이 행정/청소/물류 담당
- **길드 "관리부"**: 장비 정리, 던전 진입 전 준비, 사후 처리
- **민간 파견**: 각성 능력을 일반 산업에 활용 (감지 능력 → 탐사, 미세 능력 → 정밀 작업)
- **무능력자 커뮤니티**: F급끼리의 자조 모임, 정보 공유 (인터넷 카페/포럼)

## 4. IT 업계 vs 헌터 사회 구조 대비

| 항목 | IT 업계 | 헌터 사회 | 서사적 겹침 |
|------|---------|----------|-----------|
| 계층 | 원청→1차→2차→프리랜서 | S→A→B→...→F | 하위에서 시작, 계층 돌파 |
| 평가 기준 | 연차, 포트폴리오, 코딩테스트 | 마나량, 스킬 등급, 레이드 실적 | 숫자로 환원되는 인간 가치 |
| 착취 구조 | 단가 깎기, 야근 강제 | 위험 던전 하청, 드롭 아이템 상납 | 위험은 하위가, 이익은 상위가 |
| 이직/이탈 | 스타트업/대기업 이직 | 길드 이적, 솔로 전환 | "더 나은 곳" 가려는 욕구 |
| 비주류의 무기 | 오픈소스, 사이드 프로젝트 | 숨겨진 스킬, 비정규 전투법 | 시스템 밖의 실력 증명 |

## 5. 서사 활용 제안

### 5-1. 안강해의 배경 디테일
- 중소 SI 2차 협력사 4년차. 대기업 빌딩에 파견, 원청 사원과 같은 식당 이용 불가. 각성 후에도 F급이라 헌터 사회에서도 같은 구조 반복 → "두 세계 모두에서 하청인 남자"
- 디버깅 능력은 IT에서 "찾아내는 눈"으로 단련됨. 남들이 못 보는 버그를 찾듯, 남들이 못 보는 스킬 코드의 결함을 찾음

### 5-2. IT 용어의 전투 장면 활용
- 스킬 코드 수정 장면에서 실제 IDE 화면처럼 묘사 (변수명, 함수 호출, 에러 메시지)
- "런타임 에러" = 스킬 발동 실패, "널 포인터" = 스킬 대상 없음, "무한 루프" = 스킬 폭주
- 리팩토링 = E급 스킬을 구조 개선하여 B급 효율로 만듦 (등급은 그대로지만 성능은 다름)

### 5-3. 조직 갈등의 이중 구조
- IT 파견사에서의 갑을 경험이, 길드 내 F급 차별과 정확히 겹침
- 원청 PM = 길드 지휘관, 영업 = 길드 영입 담당, 프리랜서 = 솔로 헌터
- 안강해가 두 세계의 구조적 유사성을 자조적으로 인식하는 내레이션 가능: "하도급은 세계를 넘어도 하도급이다"

### 5-4. 무능력자 부서 소재
- F급 모임의 자조적 분위기 (IT 개발자 커뮤니티의 "야근 자랑"과 유사)
- 무능력자도 각성자이므로 미세한 능력 보유 → 안강해는 그 미세한 능력의 "코드"를 볼 수 있어 차이 발생
- 무능력자 부서에서 만난 동료가 나중에 핵심 파티원이 됨 (기존 헌터물 차별화 포인트)
